# Contributing to PSYCHO

Thank you for helping the road get stranger.

PSYCHO is a small audio-reactive game, so even modest contributions can have a
visible effect. Bug fixes, platform reports, performance improvements, tests,
documentation, accessibility work, and new musical ideas are all welcome.

All participation is governed by the
[Contributor Covenant Code of Conduct](CODE_OF_CONDUCT.md). Please read it
before joining discussions or submitting work.

## Before you start

For a small, focused fix, opening a pull request directly is fine. For a large
feature, analyzer change, new dependency, or redesign, please open a
[GitHub issue](https://github.com/skorotkiewicz/psycho-odin/issues) first. A
short conversation can prevent two good ideas from pulling the game in opposite
directions.

Keep these project principles in mind:

- The song should shape the road, speed, mood, traffic, and motion—not merely
  decorate a generic level.
- The same audio and configuration should produce the same course.
- Fast sections must remain readable and playable.
- Visual and headphone effects must remain optional and adjustable.
- Small native builds and short dependency chains are features.
- The game should keep working without a network connection or launcher.

## Set up a development build

You need Git, [just](https://github.com/casey/just), `curl`, `tar`, `jq`,
and the [platform linker required by Odin](https://odin-lang.org/docs/install/).
The bootstrap recipe currently supports Linux and macOS on AMD64 and ARM64.

```sh
git clone https://github.com/YOUR-NAME/psycho-odin.git
cd psycho-odin
just setup
just build-debug
./psycho ~/Music/a-test-song.mp3
```

`just setup` installs the project-local Odin compiler and formatter under
`~/odin-dev`. If you manage Odin yourself, build directly with a recent
toolchain that includes the raylib vendor package:

```sh
odin build src -out:psycho -debug -o:none
```

Raylib supports WAV, MP3, OGG, and FLAC. Please use audio you are allowed to
share when attaching a reproduction file to an issue.

## Find your way around

| Path | Responsibility |
| --- | --- |
| `src/main.odin` | CLI, game lifecycle, raylib setup, input, and the main loop |
| `src/audio_analysis.odin` | Song analysis and map-cache format |
| `src/track_generation.odin` | Turning musical features into course movements |
| `src/track_geometry.odin` | Closed-loop 3D centerline and road geometry |
| `src/gameplay.odin` | Steering, scoring, collisions, HUD state, and ghost rules |
| `src/render_world.odin` | Road, camera, ship, traffic, and world rendering |
| `src/render_background.odin` | Music-reactive sky and background effects |
| `src/render_ui.odin` | Ride HUD, maps, pause state, and result screen |
| `src/audio_fx.odin` | Real-time stereo/headphone processing |
| `src/config.odin` | Configuration defaults, parsing, and result persistence |
| `src/self_test.odin` | Deterministic regression checks |
| `assets/` | README and source artwork |
| `docs/` | Static project website and its local asset copies |

[MORE.md](MORE.md) explains the analyzer, track model, gameplay rules, audio
effect, and comfort decisions in more depth.

## Make a focused change

Create a branch with a descriptive name:

```sh
git switch -c fix/camera-preview
```

Prefer one concern per pull request. Avoid committing generated files:

- `psycho` or `psycho.exe`;
- `.psycho_cache/`;
- `.games/`;
- personal music or test recordings.

Use `odinfmt` through the project recipe instead of formatting by hand:

```sh
just fmt
```

Write comments for decisions and constraints, especially musical mappings and
camera math. Comments that only repeat the code are unnecessary.

## Project-specific engineering rules

### Audio thread

`audio_fx` runs inside the audio callback. Do not allocate memory, perform
file I/O, print logs, or take blocking locks there. Move setup and state changes
outside the callback, and keep shared parameter updates simple.

Run the address-sanitized test after changing buffers, pointers, channel
handling, or callback state:

```sh
just test-asan
```

### Frame loop

Avoid avoidable allocation and file I/O in the render loop. Test resizing,
fullscreen, pausing, finishing a ride, and both keyboard and mouse steering
when changing lifecycle or input code.

The player must not steer while paused, and hiding the ride HUD must never hide
the end screen.

### Analysis and map cache

Analyzer and track-generation changes should remain deterministic. Add a
focused assertion to `src/self_test.odin` when the behavior can be expressed
without a window or audio device.

Increment `MAP_VERSION` in `src/audio_analysis.odin` when a change makes old
cached maps incompatible or semantically stale. Otherwise, users may continue
loading a course generated by older rules.

Use `--analyze` to exercise the offline path without opening a window:

```sh
./psycho --analyze ~/Music/a-test-song.ogg
```

### Configuration and controls

When adding a configuration option, update all relevant places:

1. `Game_Config`, its default, parser, and clamp in `src/config.odin`;
2. `config.toml`;
3. configuration parsing assertions in `src/self_test.odin`;
4. [README.md](README.md) and [MORE.md](MORE.md), when user-facing.

When adding or changing a key, update the in-game control hint and the README
controls table. Make sure the key behaves sensibly while riding, paused, and on
the result screen.

### Visuals, motion, and comfort

The visual identity is high-contrast neon, but readability comes first:

- keep the ship, road edge, traffic, and upcoming geometry visible at maximum
  speed;
- avoid rapid full-screen flashing;
- respect the existing visual-effect toggle and strength control;
- check common window sizes, including the 800 × 600 minimum;
- include a screenshot or short recording with visual pull requests.

For website changes, check desktop and mobile layouts, keyboard focus, and
`prefers-reduced-motion`. The website keeps local copies under `docs/assets/`;
update them when their source files under `assets/` change.

### Headphone effect

Keep the binaural/stereo effect optional, conservative in level, and free of
medical claims. Test stereo and mono input, toggling while music is playing,
strength ramps, and output headroom.

## Verify before opening a pull request

The normal local gate is:

```sh
just fmt
just lint
just test
just build
```

`just test` builds a debug binary and runs the deterministic self-test. It
covers audio-effect ramps and headroom, analysis and tempo response, generated
grades and turns, loop closure, camera continuity, road visibility, controls,
configuration, and gameplay rules.

Also play the affected behavior manually. For analysis, camera, or
track-generation work, use at least two contrasting songs—for example, one
quiet or slow track and one dense or fast track. Mention what you tested in the
pull request.

Run `just test-asan` for memory-sensitive work. Run `just build-size` when a
change could materially affect the tiny release binary.

## Report a bug

Please include enough information to reproduce it:

- operating system, architecture, and whether Wine or a virtual machine is
  involved;
- release version from `./psycho --version`, or your commit hash;
- the exact command you ran;
- audio format, channel count, sample rate, duration, and the approximate
  timestamp where the issue occurs;
- relevant `config.toml` values;
- complete terminal output;
- a screenshot or short recording for visual problems;
- whether the map was newly analyzed or loaded from `.psycho_cache/`.

If you cannot share the audio, describe its structure and try to reproduce the
problem with a freely shareable or generated test file.

## Open a pull request

A useful pull request description answers four questions:

1. What changed?
2. Why is this the right behavior for PSYCHO?
3. How was it verified?
4. What should a reviewer pay special attention to?

Include screenshots for rendering, UI, website, or camera changes. Note any
platform you could not test. Keep unrelated cleanup in a separate pull request
so regressions are easier to identify.

Do not update `VERSION` in ordinary pull requests. Release versioning and tags
are maintained separately.

## Releases

This section is for maintainers.

1. Update [VERSION](VERSION) with a semantic version.
2. Run the full verification gate.
3. Commit the release changes on `main` and ensure the working tree is clean.
4. Run `just add-tag`.

The tag triggers native Linux AMD64, Linux ARM64, macOS Intel, macOS Apple
Silicon, and Windows AMD64 builds.
